## Architektur-Eckpunkte

- **Monorepo**: `apps/api` (Express, Prisma, Zod) · `apps/mobile` (Ionic Vue, Pinia, vue-i18n) · `packages/shared` (Zod-Schemas/DTOs/Enums) · `e2e` (Playwright)
- **Zentrale Abstraktion** `PortfolioSource`: jede Verbindung/jeder Import/manuelle Topf ist eine Quelle; Holdings hängen immer an genau einer Quelle (Herkunft bleibt sichtbar)
- **Provider-Interface + Registry**: 6 echte Provider; `FAKE_PROVIDERS`/`FAKE_PRICES` (nur `APP_ENV=local`) liefern deterministische Werte für Tests
- **Sicherheit**: Secrets AES-256-GCM-verschlüsselt, API liefert nur `…1234`-Preview; Ownership überall via `userId` aus dem JWT (404 statt 403, getestet); Rate-Limiting auf `/auth`
- **Beträge**: durchgehend String/`Prisma.Decimal`/BigInt — kein float in der Geld-Pipeline
- **Lokale Ports**: API **3010**, App **5173**, Postgres **5434** (5432/5433 sind auf diesem Host belegt); E2E isoliert auf 3011/5174 mit DB `crypto_tracker_e2e`, Integrationstests auf `crypto_tracker_test`

## Bekannte Einschränkungen / bewusste Entscheidungen

- **CoinGecko-Rate-Limit (ohne API-Key)**: Markt-Tab und Wertverlauf (`/portfolio/history`, 1 `market_chart`-Call je Asset) treffen den Free-Tier-Limit schnell. Der CoinGecko-Client ist gehärtet (`cgFetchJson`: Nicht-JSON/Fehler → sauberer 502 statt 500; Markt + Chart liefern bei Ausfall den letzten Cache-Stand; History überspringt fehlschlagende Assets statt komplett zu scheitern). Echter Fix gegen das Limit: kostenloser `COINGECKO_API_KEY` (in `.env.example` dokumentiert)
- **Live-Happy-Path der Exchanges ungetestet** — alle 11 Exchanges sind per Fixtures + Signatur-Known-Answer-Tests verifiziert; der Erfolgsfall braucht echte read-only Keys. Earn/Margin/Futures sind jetzt abgebildet (Binance/OKX/Bybit/KuCoin + Kraken-Suffixe; Multi-Konto über `fetchAccount`), Kraken-/KuCoin-**Futures** (eigener API-Host) bleiben zurückgestellt; die Sub-Endpoint-Parser sind mangels Keys nur fixture-/unit-getestet. Chain-Provider: Blockchair (LTC/DOGE) hat ein Tageslimit (~1440 Calls), EVM-/Tron-Token-Listen sind kuratiert (kein Indexer)
- **API-/Wallet-Sync liefert keine Kauf-/Handelshistorie**: Exchange-Provider geben nur einen Bestands-Snapshot zurück (`fetchBalances` → `RawBalance[]`), keine BUY-/Trade-/Order-Historie; Wallet-Sync erzeugt höchstens `STAKING_REWARD`-Transaktionen, aber keine BUYs. Folge: synchronisierte Bestände haben keine Kostenbasis und erscheinen **nie** im PnL (`pnl.service.ts` überspringt Snapshot-Quellen bewusst). Kostenbasis/PnL nur über manuelle Transaktionen (`POST /transactions`, Typ BUY mit `pricePerUnit`) oder CSV-Transaktions-Import
- **Transaktions-Importe**: gespeichert + Netto-Bestände (BUY/DEPOSIT − SELL/WITHDRAWAL); Kostenbasis/Gewinne rechnet der Steuerreport (`/tax/report`), laufende PnL-Anzeige im Portfolio fehlt weiterhin
- **Steuerreport**: nur Quellen mit Transaktionshistorie (CSV-Transaktionen + manuelle Transaktionen); Quellen mit reinen Bestands-Snapshots werden als „nicht enthalten" ausgewiesen. DE rechnet wallet-bezogenes FIFO (BMF 10.05.2022): verknüpfte Transfers übertragen die Kostenbasis, unverknüpfte Auszahlungen verlieren sie (Warnung). AT nutzt weiterhin globale Pools (dokumentierte Vereinfachung), verknüpfte Transfers sind dort neutral. Als Tausch verknüpfte Krypto-zu-Krypto-Vorgänge (`SwapLink`) sind in AT steuerneutral (§27b, Kostenbasis von A wandert auf B), in DE eine Veräußerung. Annahmen: Altvermögen zuerst (AT), Netzwerkgebühr-Basis verfällt still; unverknüpfte Tauschvorgänge und FX-Umrechnung nicht abgebildet. CoinGecko-Backfill historischer Kurse nur ~365 Tage zurück (Free Tier), Cap pro Lauf (40, mit Demo-Key 150) mit Hinweis-Warnung. PDF-Export bewusst immer Deutsch (Empfänger Finanzamt/Steuerberater)
- **Asset-Mapping ist global** (Assets nutzerübergreifend); deshalb nur für unmapped Assets erlaubt
- **Refresh-Token**: nativ verschlüsselt im Keychain/Keystore (erledigt); im Web weiterhin localStorage-Fallback — vor Web-prod-Deployment auf httpOnly-Cookies umstellen
- **Quellen-Umbenennen**: Backend (`PATCH /sources/:id`) + UI (Alert-Dialog auf der Quellen-Seite; Fehler-Toast bei fehlgeschlagenem PATCH statt stiller Rückkehr)
- **Solana-Spam**: Dust-/Spam-Filter aktiv — unbekannte Mints werden standardmäßig übersprungen (`includeUnknownTokens`, Default aus); bei explizitem Opt-in als unmapped importiert. Anzeigesymbol unbekannter Mints nutzt 6+6 Base58-Zeichen (Kollision praktisch ausgeschlossen); exakte Identität per Mint bräuchte eine Mint-Spalte am Asset (zurückgestellt, da preislose Dust-Tokens)
- Server-Fehlertexte sind Deutsch; das Frontend lokalisiert über die stabilen `error.code`s (die wichtigsten Codes in allen 6 Sprachen)
- **Queue-Modus in CI**: der reine BullMQ-Transport (Worker-Prozess, Repeatable Jobs) braucht Redis und läuft in CI weiterhin im Inline-Modus; die Service-Logik ist aber abgedeckt — Stale-Run-Reaper (`reapStaleRuns`, Schwelle `REAP_STALE_MS`=30 min, deutlich über dem 2-min-Neustart-Gate, damit langsame echte Syncs nicht fälschlich gereapt werden) und atomarer RUNNING-Lock (`SELECT … FOR UPDATE` pro Quelle) per Integrationstest; exklusiver Ausführungs-Lease über `SyncRun.heartbeatAt` (`executeSyncRun` beansprucht den Run atomar per geschütztem `updateMany` und schlägt ihn alle 15 s fort, `HEARTBEAT_STALE_MS`=45 s) — verhindert doppelte Side-Effects bei verlorenem Enqueue-ACK und blockiert einen zweiten Start auch jenseits des 2-min-Gates, solange der Heartbeat lebt; ein toter (alter/fehlender) Heartbeat gibt die Quelle frei; Enqueue dedupliziert per `jobId=runId` + Backoff-Retry für Infra-Fehler; bei totem Redis fällt `requestSync` per `pingRedis` (2-s-Timeout, gecachter Client) bewusst auf Inline zurück (kein verwaister RUNNING-Run, getestet in `sync-redis-fallback.integration.test.ts`)
- **Deployment Worker-Prozess**: bei gesetztem `REDIS_URL` ist der Worker (`apps/api` `start:worker`) Pflicht — sonst bleiben Jobs liegen. Deklariert in `Procfile` (`web`/`worker`/`release`); lokal `pnpm dev:queue` (API+Worker+App). `docker-compose.yml` stellt nur Infra (Postgres/Redis), nicht die App
- **CSV-Preset persistiert**: das erkannte Preset (KRAKEN/BITPANDA) wird beim Upload auf `CsvImport.preset` gespeichert und beim Confirm verwendet — keine Header-Re-Detection mehr, entkoppelt von manuellen Mapping-Overrides
- **Kraken-Ledger-Preset**: vollständig verarbeitet — vorzeichenbehaftete Beträge (Abflüsse negativ), nur die richtungsneutralen Typen `trade`/`spend` werden über das Vorzeichen in BUY/SELL aufgelöst (`receive`/`deposit`/`withdrawal`/`staking`/`transfer` bleiben als eindeutige Aliase erhalten), unbekannte/vertippte Ledger-Typen werden **nicht** geraten, sondern als Fehlerzeile abgewiesen; XXBT/ZEUR-Normalisierung via `normalizeKrakenAsset`, Fiat-/Fee-Legs verworfen; Row-Level-Tests für Kraken **und** Bitpanda. CSV-Zeitstempel ohne Zone werden deterministisch als Europe/Berlin interpretiert (server-TZ-unabhängig), damit der Steuerreport an Tagesgrenzen nicht ins falsche Zivildatum kippt
- **Staking-Reward-Import, bewusste Grenzen**: Solana-Erst-Import max. 30 Epochen zurück (RPC-Pruning; ältere Historie via CSV); ETH-Validator-Rewards nur mit beaconcha.in-Key (v1-API verlangt inzwischen einen — Antwortform gegen Live-API noch unverifiziert, Fixtures nach v1-Doku); Exit-Withdrawals ≥ 8 ETH werden als Principal übersprungen (Heuristik); stETH-Rebase-Erträge und MEV/Tips nicht erfasst (Disclaimer). ERC-20 nur kuratierte Liste (kein Indexer)
